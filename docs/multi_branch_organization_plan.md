# 다매장(조직) 서비스 계획서

**문서 목적:** 개인(단일) 매장은 기존과 동일하게 운영하고, 본사·다매장(슈퍼 다지점) 사용자를 어떻게 받을지·묶을지·요금·관리 화면을 어떻게 만들지 정리한다.  
**최종 수정:** 2026-04-20

**플랫폼 결정(요약):** 다매장·비교·취합이 필요한 고객에는 **조직(본사) 단위 모델을 넣는 방향**으로 한다. (한 사용자에게 테넌트만 여러 개 연결하는 방식보다 운영·과금·RLS·비교 UI가 명확해진다.)

---

## 1. 범위와 원칙

### 1.1 개인 매장(단일 테넌트)

- **변경 없음:** 기존 `tenant_id` 단위로 주문·CRM·재고·설정 등 전 기능을 그대로 사용한다.
- 가입·온보딩·구독·RLS는 현행 모델을 유지한다.

### 1.2 앱 관리 차원에서 본사(조직) 개념을 두는 이유

- **지점 비교·취합:** 매출·주문·지표를 “같은 그룹 안의 지점들”끼리만 비교하려면, 그 그룹의 경계가 코드·DB·과금에서 일치해야 한다. **조직(organization)** 이 그 경계다.
- **권한과 감사:** 누가 어떤 지점 묶음을 볼 수 있는지 `organization_members`와 RLS로 한 번에 표현할 수 있다. “계정 두 개로 두 매장 쓰기”보다 **본사 담당자 1계정 + 소속 지점 N개**가 설명 가능하다.
- **과금:** 본사 모듈료·포함 지점 수를 조직 단위로 붙이기 쉽다.
- **정리:** 동일 대표가 개인적으로 매장만 2개 운영하는 경우에도, **비교·본사 화면을 쓰게 할 계획이면** 조직 하나에 지점 2개를 넣고 본사 사용자를 붙이는 편이 앱 관리에는 유리하다. (비교 기능을 쓰지 않고 완전 분리만 원하면 기존처럼 매장별 계정으로도 가능하나, **플랫폼 기본 방향은 다매장 니즈 = 조직 개설**로 둔다.)

### 1.3 슈퍼 다지점 사용자(조직 사용자)

- **정의:** 플랫폼 관리자가 지정한 **조직(organization)** 아래에 여러 **지점(기존 `tenants` 행)** 이 묶여 있고, 해당 조직에 권한이 부여된 사용자.
- **권한의 출처:** 지점을 스스로 “묶지” 않는다. **반드시 앱(플랫폼) 관리자**가 조직을 만들고 지점을 배정하고, 조직 관리자 계정을 연결한다.
- **데이터 접근:** 배정된 지점들에 대해 정책이 허용하는 범위에서 **열람·비교·(단계적으로) 본사 관리**가 가능하다.

## 2. 운영 프로세스: 신청 → 검토 → 배정

### 2.1 신청은 어떻게 받을까 (권장 흐름)

| 단계 | 주체 | 내용 |
|------|------|------|
| A. 신청 | 고객(예정 본사) | 공식 채널에서 “다매장·본사 패키지” 문의 또는 앱 내 **신청 폼**(선택) 제출: 상호, 사업자번호, 대표 연락처, **포함 예정 지점 수·지점명 목록**, 현재 각 지점 Floxync 사용 여부 |
| B. 검토 | 플랫폼 CS/영업 | 계약·과금 단위 확정, 지점이 이미 가입돼 있는지 확인, 법적 동의(지점 데이터 본사 열람) 필요 시 별도 동의서 |
| C. 개설 | 플랫폼 관리자(`super_admin`) | `organizations` 생성, 지점(`tenants`) 매핑, `organization_members`에 **조직 슈퍼 사용자** 등록 |
| D. 온보딩 | 조직 사용자 | 초대 메일/임시 비밀번호 또는 매직링크로 로그인 → 비밀번호 변경 → **본사 대시보드** 진입 |

**앱 내 신청 폼이 없어도 됨:** 초기에는 이메일·노션·구글폼으로 받고, 관리자만 대시보드에서 조직을 만드는 방식으로 시작할 수 있다. 트래픽이 늘면 `/dashboard` 또는 랜딩에 “다매장 도입 문의”를 두고, 나중에 DB에 `organization_requests` 테이블로 정식화한다.

### 2.2 지점을 어떻게 묶어줄까 (관리자 작업)

1. **플랫폼 관리 전용 화면** (예: `전국 화원사 관리` 인근 또는 하위 메뉴 `조직 관리`)
   - 조직 생성: 이름, 메모, 상태(활성/중지)
   - **지점 추가:** 기존 `tenants` 목록에서 검색 → 조직에 연결 (`organization_tenants` 또는 `tenants.organization_id`)
   - **지점 제거:** 계약 종료 시 매핑만 해제(테넌트 데이터는 삭제하지 않음이 기본)
2. **사용자 연결**
   - 조직당 1명 이상 `org_admin`(슈퍼 다지점) 지정
   - (선택) `org_viewer` 등 읽기 전용 역할로 확장
3. **감사**
   - 누가 언제 어떤 지점을 조직에 넣었는지 로그(최소: 관리자 계정 + 타임스탬프)

**주의:** 지점 실무 직원의 `profiles.tenant_id`는 그대로 두고, 본사 담당자만 별도 계정으로 `organization_members`에 넣는 방식이 RLS·혼선이 적다. (본사 담당이 동시에 한 지점의 점주인 경우는 한 사용자에 tenant + org 멤버십 병행 가능 — 설계 시 허용 여부 명시)

## 3. 요금(슈퍼 다지점) 책정 가이드

법적·회계 자문은 별도이며, **제품 관점에서의 책정 프레임**만 적는다.

### 3.1 과금 단위 후보

| 모델 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **조직 월 정액 + 지점당 과금** | 본사 기능료 + 지점 수 × 단가 | 매출 예측 쉬움, 지점 늘면 자연 증가 | 지점 수 협의 필요 |
| **포함 지점 N개 패키지** | 예: 최대 5·10·무제한 티어 | 견적 단순 | 실제 지점과 안 맞으면 조정 필요 |
| **조직 단독 맞춤 견적** | 연간 계약 | 대형 프랜차이즈 대응 | 운영 공수 |

### 3.2 “얼마가 적당한가”에 대한 현실적인 시작점

- **기준:** 지점당 **이미 내는 Floxync 구독(pro/erp 등)** 은 유지하고, 그 위에 **본사 레이어 요금**을 올리는 이중 구조가 이해하기 쉽다.
- **볼륨 가이드(예시, 실제는 원가·경쟁사·LTV 반영해 조정):**
  - **조직 기본료:** 월 5만~15만 원대 — 비교 대시보드·지점 배정·계정 1~2개
  - **지점 슬롯:** 지점당 월 1만~3만 원 **추가** 또는 패키지에 N개 포함
  - **대형(10지점+):** 월 30만~80만+ 또는 연 계약 할인

문서에 고정 금액을 박아두기보다, **“지점 구독 + 본사 모듈료”** 두 줄로 견적서를 쓰고, 파일럿 1~2곳에서 실제 사용 지표를 본 뒤 가격을 고정하는 것을 권장한다.

### 3.3 결제 시스템 연동

- 초기: **수동 청구**(계좌·세금계산서) + 관리자가 `tenants`/`organizations`에 `billing_tier` 또는 만료일 플래그 설정
- 이후: 토스 등 기존 결제와 **“조직 구독”** 상품 코드 추가

## 4. 슈퍼 다지점 사용자의 관리 페이지

### 4.1 진입과 네비게이션

- 로그인 사용자가 `organization_members`에 있고 역할이 `org_admin`(또는 동급)이면 사이드바에 **「본사·다매장」** 섹션 표시
- 개인 매장 메뉴와 **병행 표시 여부:** 본사 담당이 특정 지점 `tenant_id`도 가지고 있으면 기존 지점 메뉴 + 본사 메뉴 모두 노출

### 4.2 화면 구성(페이지 제안)

| 페이지 | 목적 | 주요 기능 |
|--------|------|-----------|
| **본사 개요** `/dashboard/hq` | 한눈에 상태 | 소속 지점 카드, 계약/구독 요약(관리자가 넣은 메타), 알림 |
| **지점 비교** `/dashboard/hq/branches/compare` | 경영 분석 | 기간 선택, 지점별 매출·주문 건수·평균 객단가 등(기존 analytics/reports 쿼리 재사용) |
| **지점 상세(읽기)** `/dashboard/hq/branches/[tenantId]` | 특정 지점 깊이 보기 | 요약 지표, 최근 주문·고객 수 등 — **RLS 또는 서버 집계로 허용된 tenant만** |
| **지점 업무 모드(전환)** | 실무자와 동일 화면 | 지점 선택 후 “이 지점으로 업무 보기” → 세션/쿠키에 `active_tenant_id` 설정, 기존 ERP 화면은 그 tenant 기준으로 동작(감사 로그 권장) |
| **본사 설정(Phase C)** | 조직 단위 | 공지 작성, 목표 KPI, (선택) 승인 대기함 — 초기에는 비우고 로드맵만 둘 수 있음 |

### 4.3 단계별로 “무엇이 가능한가”

**Phase B (MVP — 구현 1차)**

- 소속 지점 목록 조회
- 기간·지표별 **비교 표/차트** (읽기 전용 집계)
- **지점 컨텍스트 전환** 후 기존 주문·CRM 등 화면 이용(권한은 org 정책과 동일하게 제한)

**Phase C (확장)**

- 조직 단위 공지·자료실
- 지점별 목표·전략 메모(본사만 편집)
- 일부 데이터에 대한 **읽기 전용 공유 링크**(보안 검토 후)
- `org_viewer` 등 세분화 역할

**불가(또는 별도 계약)로 두는 것이 안전한 것**

- 다른 조직 지점 보기
- 플랫폼 전역 설정·타 조직 요금 변경
- `super_admin` 전용 메뉴(테넌트 생성·전체 과금) — 조직 사용자에게 노출하지 않음

## 5. 기술 전제 (요약)

- **데이터:** `organizations`, 조직↔지점 매핑, `organization_members`; 지점 데이터는 기존 `tenant_id` 스키마 유지
- **RLS:** `org_admin`에게 “소속 `tenant_id` 집합”에 대한 SELECT(및 정책에 따른 UPDATE) 허용; 또는 MVP에서 집계만 **서버 API + 권한 검증**으로 제한
- **앱 변경:** [dashboard/layout.tsx](d:\mapp\floxync-saas\src\app\dashboard\layout.tsx) 역할 분기, 사이드바에 HQ 섹션, 관리자용 조직 CRUD 화면

### 5.1 코드 반영 현황 (MVP)

- **DB 스크립트:** [supabase/organization_schema.sql](d:\mapp\floxync-saas\supabase\organization_schema.sql) — Supabase SQL Editor에서 실행 후 사용.
- **본사 공지 테이블:** [supabase/organization_announcements_schema.sql](d:\mapp\floxync-saas\supabase\organization_announcements_schema.sql) — 조직 스키마 적용 후 실행.
- **플랫폼 관리:** `/dashboard/admin/organizations` — 조직 생성, 지점(`tenants`)에 `organization_id` 연결, 본사 사용자 이메일로 멤버 추가(API가 `org_admin` 갱신).
- **본사 화면:** `/dashboard/hq` — 지점별 매출·객단가, 일자별 스택 차트, 배송/픽업/매장 수령 카드, 취소 비율, 본사 공지 작성·목록. 집계는 `/api/hq/summary`. 공지 API는 `/api/hq/announcements`. **`.env`에 `SUPABASE_SERVICE_ROLE_KEY` 필요(집계).**
- **지점 배너:** 소속 조직 공지는 일반 대시보드 상단([dashboard-main](d:\mapp\floxync-saas\src\components\layout\dashboard-main.tsx) + [OrgAnnouncementBanner](d:\mapp\floxync-saas\src\components\hq\org-announcement-banner.tsx))에 표시(`/dashboard/hq`에서는 중복 방지로 숨김).
- **기존 매장:** `organization_id`가 NULL이면 동작·RLS와 동일한 단일 매장 모델.

## 7. 용어 정리

| 용어 | 의미 |
|------|------|
| 개인 매장 | 사용자 1명(또는 직원) ↔ 테넌트 1개, 기존 모델 |
| 지점 | `tenants` 테이블의 한 행(매장 단위) |
| 조직 | 본사·브랜드 단위로 여러 지점을 묶는 논리적 그룹 |
| 슈퍼 다지점 사용자 | 조직에 부여된 `org_admin` 등, 묶인 지점 데이터를 본사 관점에서 다루는 사용자 |
| 플랫폼 관리자 | `super_admin`, 조직·지점 배정의 유일한 운영 주체(초기 설계) |
