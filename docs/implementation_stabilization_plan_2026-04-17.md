# Florasync SaaS 수정 계획서 (테스트 단계 경량 버전)

작성일: 2026-04-17  
적용 범위: `src/app`, `src/components`, `src/stores`, `supabase`, `docs`

## 1) 목표

현재는 테스트 단계이므로, 운영 수준의 강한 구조 개편보다 다음 3가지를 우선한다.

1. 테스트 진행을 막는 오류 제거
2. 위험한 포인트를 "완전 개편" 대신 "안전 가드"로 완화
3. 검증 결과를 빠르게 반영할 수 있는 가벼운 작업 단위 유지

본 계획서는 "최소 수정 + 빠른 검증"을 기본 원칙으로 한다.

## 2) 핵심 이슈 요약

### P0 (지금 테스트에 영향 큰 항목)

- 하드코딩된 관리자 승격 로직 존재 (테스트용 예외 여부 확인 필요)
  - 대상: `src/stores/auth-store.ts`, `src/app/dashboard/layout.tsx`
  - 리스크: 특정 이메일 기반 권한 우회, 감사 추적 어려움

- 서버 API의 키 폴백 로직 위험 (운영 전환 전 정리 필요)
  - 대상: `src/app/api/pos/webhook/[provider]/route.ts`, `src/app/api/delivery/webhook/kakao/route.ts`
  - 리스크: `service_role` 미설정 시 `anon` 사용으로 권한/동작 불일치

- 결제 confirm 멱등성 부족 (중복 호출 최소 가드 필요)
  - 대상: `src/app/api/payments/confirm/route.ts`
  - 리스크: 중복 승인/재시도 시 테넌트 구독 상태 오염 가능

### P1 (테스트 병행 개선)

- 스키마 파일 분산 관리
  - 대상: `supabase_schema.sql`, `supabase/ai_messaging_schema_update.sql`, `supabase/support_faq_schema.sql`
  - 리스크: 환경별 누락/불일치

- AI 지식 소스의 파일 의존성
  - 대상: `src/app/api/ai/support/route.ts`, `docs/ai_knowledge_base.md`
  - 리스크: 배포 환경에서 파일 부재 시 응답 품질 저하

- 플랜/권한 제한 로직 중복
  - 대상: `src/app/dashboard/layout.tsx`, `src/components/layout/sidebar.tsx`
  - 리스크: 화면별 접근 정책 불일치

## 3) 실행 계획 (경량)

## Phase A. 테스트 진행 보장 (D+0 ~ D+2)

### A-1. 관리자 권한 로직 임시 가드
- 이메일 하드코딩은 즉시 삭제하지 않고, `TEST_MODE` 플래그로 한시적 분리
- 플래그 OFF 시 DB role 기준만 사용되도록 분기 정리
- 테스트 종료 전 제거 대상임을 코드 주석/문서에 명시

완료 기준:
- 테스트/비테스트 모드의 권한 경로가 명확히 분리됨
- 테스트 중 권한 오류로 막히는 이슈가 재현되지 않음

### A-2. 서버 키 안전 가드 추가
- `service_role` 미설정 시 즉시 실패 대신 "명확한 경고 + 제한 동작" 적용
- 위험 API만 우선 점검하고, 전체 제거는 운영 전환 단계로 이월
- 환경 변수 누락 시 어디서 실패했는지 로그 메시지 표준화

완료 기준:
- 테스트 중 원인 불명 인증 실패가 줄어듦
- 운영 전 정리할 코드 위치 목록이 문서화됨

### A-3. 결제 중복 방지 최소 적용
- confirm API에 중복 요청 가드(동일 `paymentKey` 빠른 차단) 추가
- 전체 트랜잭션 개편 대신, 중복 호출 시 무해 응답하도록 우선 처리
- `orderId` 파싱 실패 시 사용자 친화 에러 메시지 제공

완료 기준:
- 같은 결제를 여러 번 호출해도 구독 상태가 깨지지 않음
- 테스트 로그에서 결제 실패 원인 분류 가능

## Phase B. 테스트 병행 리팩토링 (D+3 ~ D+7)

### B-1. 스키마 적용 순서 문서화 우선
- 분산 SQL을 즉시 통합하지 않고, 우선 실행 순서 문서부터 고정
- 테스트 환경 재현에 필요한 최소 스크립트 세트 정의
- 마이그레이션 완전 통합은 운영 전환 단계에서 진행

완료 기준:
- 신규 테스트 환경을 같은 순서로 재현 가능
- 누락 스크립트로 인한 QA 실패 감소

### B-2. AI 지식 소스 경고 체계 정리
- 파일/DB 소스 우선순위는 유지하되, 누락 시 명확한 fallback 로그 제공
- 답변 품질 저하 상황을 테스트에서 체크 가능하도록 기준 문구 정의
- 구조 개편은 보류하고 모니터링 포인트만 먼저 넣기

완료 기준:
- 지식 소스 누락 시에도 AI 응답이 완전히 실패하지 않음
- 테스트 결과에 근거한 개편 필요성 판단 가능

### B-3. 권한/플랜 정책 중복 최소화
- 공통 모듈 완전 통합 대신, 불일치가 큰 분기부터 우선 정리
- `layout`/`sidebar` 간 핵심 조건만 맞추고 세부 리팩토링은 후순위

완료 기준:
- 주요 메뉴 노출 불일치 이슈가 테스트에서 재현되지 않음

## Phase C. 테스트 품질 체계 (D+8 ~ D+14)

### C-1. 핵심 시나리오 수동/반자동 테스트 세트
- 결제 플로우: 성공/실패/중복 confirm
- 권한 플로우: tenant_user, tenant_admin, super_admin 접근 경계
- 웹훅 플로우: 재전송/지연 도착 시 상태 정합성

완료 기준:
- 테스트 라운드마다 동일 체크리스트로 품질 확인 가능
- 자동화가 없어도 회귀 여부를 빠르게 판정 가능

### C-2. 디버깅 로그 최소 표준화
- 결제/웹훅/권한 거부 로그 구조화
- 장애 시 원인 추적 가능한 request id 또는 correlation id 추가

완료 기준:
- 테스트 실패 케이스의 원인을 로그로 1차 분류 가능
