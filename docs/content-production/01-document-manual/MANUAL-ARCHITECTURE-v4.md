> **v3와의 관계:** v3는 **두 축 구조**(상황별 S + 메뉴 M)와 언어 원칙을 정의했습니다.  
> **v4는 Track M(메뉴 백과)의 깊이·범위·제작 방법**을 재정의합니다.  
> **정본 HTML:** `docs/floxync-manual.html` → `/docs/manual`  
> **작성일:** 2026-07-04  
> **전제:** 본사(5장)는 **후순위**. 일반 매장 사용자가 매일 보는 화면부터 **버튼 단위**로 완성합니다.

## 0. 단일 페이지 원칙 (필수)

**백과사전(전체 메뉴) · 상황별 가이드 · FAQ · 부록(앱 받기)은 모두 한 HTML 안에 있어야 합니다.**

| 항목 | 내용 |
|------|------|
| **URL** | `/docs/manual` 하나 (앱 헤더 「사용 설명서」도 여기로) |
| **파일** | `docs/floxync-manual.html` (배포 시 `pack/.../docs/` 동기화) |
| **이동** | 페이지 새로고침 없이 **앵커** (`#sit-hub`, `#nav-orders`, `#faq-support`) |
| **목차** | 왼쪽 **고정 사이드바** — 섹션 블록으로 구분 |

### 0.1 한 파일 안의 층 (위에서 아래)

```
floxync-manual.html  (한 페이지)
│
├── 🏠 #manual-hub          … 어디서부터 읽을지
├── 🔗 #integrations-hub    … 연동 한눈에 (저장→인쇄·장부)
├── 📚 1장 환경 설정        … #settings-*  ← 처음 필독
├── ⭐ Track S — #sit-hub    … 상황별 (급할 때)
├── 📚 2~3장 주문·CRM·매출  … #orders-*, #crm, #revenue-engine
├── 📚 4장 일상 메뉴 백과   … #nav-* (버튼 단위 확장 · v4)
├── 🎀 PR 리본 백과         … #nav-printer, #nav-ribbon-from-order (버튼 많음 · 별도 P)
├── 📚 5장 다지점·본사      … #hq-org-* (후순위)
├── 📲 부록 앱·프로그램     … #appendix-apps
└── ❓ FAQ · 고객센터       … #faq-support
```

**하지 않는 것**

- 사장님용 설명을 **메뉴별 별도 HTML/Notion/ PDF**로 나누어 링크만 모으기  
- 상황별만 다른 URL, 백과만 다른 URL  
- 길어진다고 **사용자에게 여러 탭/여러 사이트**로 쪼개기  

**길어짐 대응 (개발 편의만)**

- 원고는 `drafts/`·`manual-parts/`에 쪼개 **작성**해도 됨  
- **빌드 결과는 항상 `floxync-manual.html` 하나**로 합침  
- 사이드바·`#manual-hub`·화면 간 `#앵커` 링크로 탐색

## 1. 왜 v4가 필요한가

### 1.1 현재 매뉴얼의 갭 (사실)

| 화면 | 코드상 UI 요소(버튼·필드·메뉴) 추정 | 현재 `#nav-*` 분량 | 충족도 |
|------|--------------------------------------|-------------------|--------|
| 업무 홈 `/dashboard` | ~18 | 3문단 | **~15%** |
| 새 주문 `/orders/new` | ~45 | 5문단 | **~10%** |
| 모바일 주문 `/orders/new-mobile` | ~40 | 거의 없음 | **~0%** |
| 주문 현황 `/orders` | ~35 | 8문단(일부 상세) | **~25%** |
| 환경 설정 `/settings` | ~55 (9탭) | 1장 분산·일부 탭만 | **~30%** |
| 배송·픽업 `/delivery` | ~22 | 2문단 | **~10%** |
| 고객·상품·지출 등 | 각 15~20 | 1~2문단 | **~10%** |
| **테넌트 메뉴 전체** | **~280+** | 요약 위주 | **부족** |

요즘 사용자는 **짧은 글도 안 읽습니다**. 대신 **「이 버튼 누르면 뭐야?」**를 검색·스크롤로 찾습니다.  
그래서 Track M은 **한 화면 = 한 챕터**, **한 버튼 = 한 항목**이 되어야 합니다. 분량이 수십 페이지·수백 항목이 되는 것은 **의도된 목표**입니다.

### 1.2 v4 목표 (한 문장)

> **FloXync 테넌트 메뉴의 모든 화면·구역·버튼·필드를 빠짐없이 설명하는 백과사전**을 만들고, 상황별 가이드(S)는 그 위에 **짧은 입구**만 둔다.

## 2. 두 축 역할 재정의

| 축 | 분량 | 역할 | 비유 |
|----|------|------|------|
| **Track S** (상황별) | 화면당 5~15분 | 급할 때 순서만 | 레시피 카드 |
| **Track M** (백과) | 화면당 **2~8페이지** | 모든 버튼·필드·예외 | 제품 설명서 전체 |

**중복 규칙**

- S에는 **순서 번호만** (버튼 이름은 앱과 동일하게)
- S 끝에 `📖 자세히: #nav-orders-new-header` 링크
- M에는 **같은 순서를 반복하지 않음** — 대신 버튼 레지스트리로 해소

## 3. Track M 표준 구조 (화면 1개 = article 1개)

모든 `#nav-{screen}` 또는 `#nav-{screen}-{zone}` 절은 아래 **고정 목차**를 씁니다.

```markdown
## [화면 이름] — [한 줄 목적]

배지: 🖥️ PC | 📱 모바일 | 플랜 제한(ERP/PRO) | 최종 검수일 YYYY-MM-DD

### 이 화면이 하는 일
(2~3문장, 사장님 말투)

### 어디서 열나
- 사이드바: …
- 경로: `/dashboard/...`
- 바로가기: …

### 화면 지도 (스크린샷 1장 + 번호 오버레이)
![annotated](manual-screenshots/nav-orders-00-overview.png)
① 상단 버튼줄 ② 통계 ③ 필터 ④ 목록 ⑤ 행 메뉴

### 구역별 상세

#### ③-1 필터: 상태
| # | UI 표기 (앱 그대로) | 하는 일 | 누르면/입력하면 | 주의 |
|---|---------------------|---------|-----------------|------|
| 1 | 전체 | … | … | … |
| 2 | 처리중 | … | … | … |

#### ⑤-2 행 더보기(⋮) 메뉴
(각 항목 3~5줄 + 관련 다이얼로그 링크)

### 다이얼로그·시트 (이 화면에서 열리는 것)
- `#nav-orders-dialog-edit` …

### 자주 묻는 말 (이 화면만)
- Q / A

### 막히면
- …

### 연결
- ⚡ 빠르게: #sit-…
- 다음 화면: #nav-…
```

### 3.1 버튼 레지스트리 필수 필드

각 버튼·토글·탭·메뉴 항목마다:

| 필드 | 설명 |
|------|------|
| **UI 표기** | `ko.json` / 화면에 보이는 글자 **그대로** |
| **위치** | 상단 / 행⋮ / 다이얼로그 / 모바일 하단 등 |
| **하는 일** | 한 문장 |
| **누르면** | 화면 전환·저장·인쇄·토스트 등 |
| **안 보일 때** | 플랜·권한·연동·모바일 숨김 |
| **관련 DB/상태** | 사장님용: 「주문 상태가 완료로 바뀝니다」 (API명 금지) |

## 4. HTML·앵커 체계 (v4)

### 4.1 네이밍

```
#nav-{menu}                    … 화면 개요 (지도 + 목차)
#nav-{menu}-{zone}             … 구역 (header, filter, table-row-menu, …)
#nav-{menu}-dialog-{name}      … 모달/시트
#nav-{menu}-mobile             … 모바일 전용 변형 (있을 때)
```

**예시 — 주문 현황**

```
#nav-orders
#nav-orders-stats
#nav-orders-header-actions
#nav-orders-filters
#nav-orders-bulk
#nav-orders-row-menu
#nav-orders-dialog-detail
#nav-orders-dialog-edit
#nav-orders-dialog-outsource
#nav-orders-mobile
```

### 4.2 사이드바 4장 목차 (확장)

기존 `#nav-map` 아래 **화면별 하위 목차**를 2단계로:

```
4. 일상 메뉴
  4-0 지도
  4-1 업무 홈
    └ 구역 4-1-a … (접이식)
  4-2 새 주문 (PC)
  4-2m 새 주문 (모바일)
  4-3 주문 현황
  …
```

HTML이 길어지면 **사용자용 URL은 쪼개지 않고**, 개발용으로만 `manual-parts/` 분할 빌드 스크립트 검토 (Phase 2). **합친 결과 = `floxync-manual.html` 단일 파일** (§0 단일 페이지 원칙).

## 6. 제작 워크플로 (코드 ↔ 문서 동기화)

### 6.1 화면당 제작 순서 (반복)

```
1. TSX 읽기 → BUTTON-INVENTORY 행 추가/갱신
2. ko.json 라벨 대조 (tenantFlows)
3. 스크린샷 TODO 등록 (구역별 PNG)
4. HTML article 골격 (#nav-*-zone)
5. 버튼 레지스트리 테이블 채우기
6. 사장님 실기 1회 → 틀린 표기 수정
7. 최종 검수일 메타 갱신
```

### 6.2 자동화 (AI + 스크립트)

| 스크립트 (예정) | 역할 |
|-----------------|------|
| `scripts/audit-manual-buttons.ts` | TSX의 Button/MenuItem label vs 매뉴얼 테이블 diff |
| `scripts/extract-tenant-labels.ts` | `ko.json` tenantFlows → 화면별 CSV |
| 기존 `generate-manual-svg-placeholders.py` | 구역별 SVG 목업 |

### 6.3 스크린샷 정책

- **구역마다 1장** (전체 + 번호 오버레이)
- 파일명: `nav-{screen}-{zone}.png`
- 목록: `SCREENSHOT-TODO-사장님.md`를 **화면×구역** 체크리스트로 확장
