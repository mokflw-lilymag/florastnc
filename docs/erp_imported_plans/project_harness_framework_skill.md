# Lilymagerp-v4 통합 개발 및 배포 하네스 (Harness)

이 스킬은 Lilymagerp-v4 프로젝트에서 에이전트가 코드를 수정하고 배포하는 전체 생명주기에 걸쳐 발생하는 **실수와 누락을 원천 차단**하기 위한 메타 프레임워크(Meta-framework)입니다. 

에이전트는 이 프로젝트에서 작업을 수행할 때 반드시 이 하네스의 체크리스트를 준수하여 스스로의 내공을 쌓고 품질을 보장해야 합니다.

## 1. UI/UX 디자인 수정 하네스
- **프론트엔드/백엔드 분리 고려:** 웹앱(`src/`)과 윈도우앱(`electron/`, `bridge-app/`) 양쪽에서 시각적/기능적 동일성이 보장되는가?
- **인쇄 템플릿(HTML) 검수:** 영수증 등 인쇄물 수정 시, `margin`, `padding`, `width` 속성이 프린터 스풀러(Chromium vs SumatraPDF)에서 다르게 렌더링될 가능성이 없는지 확인했는가? (절대 규칙: 우측 짤림 방지 및 좌측 쏠림 방지를 위해 반드시 `noscale` + HTML `width: 72mm` 고정 포맷을 유지할 것. 절대 100%로 변경하지 말 것)
- **모바일/데스크톱 대응:** 반응형 요소가 깨지지 않는지 확인했는가?

## 2. 코드 로직 및 브릿지(Bridge) 수정 하네스
- **실행 파일 종속성:** `bridge-app/` 내의 파일(JS, HTML 등)을 수정했다면, **반드시 `npx pkg .` 명령어로 `ppbridge.exe` 를 재컴파일** 하였는가?
- **[CRITICAL] 선보고 후조치 및 깃 푸시/빌드 사전 승인제:**
  - 코드 수정 완료 후 사장님의 **명시적인 승인 지시("푸시해", "빌드해")가 내리기 전에는 절대로 `git push` 또는 `npm run build` 등의 무거운 도구 명령을 스스로 먼저 가동하지 마십시오.** (기계적인 명령어 남발 및 아까운 토큰 낭비 절대 차단)
- **[CRITICAL] 로컬 교차/예외 시나리오 정밀 검증:**
  - 단순히 코드를 수정한 직후 돌아간다고 자랑하지 말고, 관련 연계 시나리오(예: 신규 상품 등록 시 중복 락은 걸리는지, 기존 상품 정보 수정 시에는 본인 이름 매치 락이 바이패스되는지, 실제 등록된 지점명 문자열이 UI 상단 선택 지점 필터와 철자 하나 틀림없이 100% 동기화되는지 등)를 로컬에서 사전에 철저히 검증하고 보고하십시오.
- **[CRITICAL] 매장 실무 중심의 주도적 제안:**
  - 지시를 수동적으로 땜질하지 말고, 실제 현장 환경(바코드 10자 이내 크기 제약, 대/중분류 키워드 조합형 3글자 바코드 등)을 파악하여 더 편리한 대안을 적극 제안하십시오.
- **로컬 테스트:** 코드를 수정한 직후 무작정 `git push` 하지 않고, 사용자에게 로컬 화면에서 테스트를 요청한 후 **"승인(합격)"을 받았는가?**

## 3. 빌드 및 배포(Release) 하네스
- **버전 관리:** 빌드 전 `package.json` 의 버전을 올바르게(Version Bump) 올렸는가?
- **Git 상태:** 새로 컴파일된 바이너리(`ppbridge.exe` 등)와 변경된 코드를 `git add .` 및 `git commit` 하여, 원격 저장소(`main` 브랜치)에 완벽히 `push` 했는가?
- **순서 엄수:** `git push` 완료 -> `npm run electron:build` 진행 -> 완료 후 `npm run release:windows` 스크립트 실행의 순서를 철저히 지켰는가?
- **NSIS 설치 충돌 방지 (필수 보장):** 사용자가 앱을 켜둔 상태에서 새 버전을 수동 설치할 때 발생하는 파일 권한 충돌 에러를 원천 차단하기 위해, `build/installer.nsh` 파일의 `customInit` 단계에서 `nsExec::Exec 'taskkill /F /IM Lilymagerp.exe'`를 호출하여 앱을 강제 종료하는 로직이 포함되어 있습니다. 빌드 설정 수정 시 이 로직이 절대 훼손되지 않도록 주의하세요.

## 4. 내공 축적 (지속적 개선)
- 작업 중 새로운 버그 패턴이나 예외 상황을 발견하면, 즉시 이 `project_harness_framework` 스킬 파일이나 `release_checklist`에 내용을 추가하여 **다시는 동일한 실수를 반복하지 않도록 기억**합니다.
