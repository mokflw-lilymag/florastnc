# 초기 기초자료 시드 유료 서비스 — 기획·개발 계획서

작성 목적: 신규 매장 온보딩 시 **상품·자재·거래처·카테고리(상품/자재/지출)** 를 표준 패키지로 일괄 세팅하는 **유료 서비스**와, 이를 뒷받침하는 **고정 시드 소스·운영·기술** 방향을 한 문서로 정리한다.

---

## 1. 배경 및 목표

- 셀프 온보딩만으로는 **빈 화면**, **분류 습관 불일치**, **초기 문의·이탈**이 발생하기 쉽다.
- 반면 매장마다 단가·SKU·실제 취급 품목은 다르므로, **완성된 실데이터를 대신 보장**하기보다 **검증된 초안(시드)** 을 빠르게 깔아 주는 모델이 현실적이다.
- 목표: **표준 세팅(유료)** 으로 시간을 팔고, 필요 시 **커스텀(고액)** 으로 매장 구조에 맞춘다. 시드 원본은 **고정·버전 관리**하여 동일 상품이 여러 고객에게 **항상 같은 구성**으로 제공되게 한다.

---

## 2. 서비스 개요

| 구분 | 내용 |
|------|------|
| 명칭(가칭) | 초기 기초자료 세팅 / 마스터 시드 패키지 |
| 제공 주체 | 관리자(super_admin)가 결제·요청 확인 후 **지정 테넌트에 일괄 반영** |
| 포함 데이터 | 상품 카테고리, 자재 카테고리, 지출 카테고리, 상품 마스터, 자재 마스터, 거래처(샘플) |
| 데이터 출처 | **릴리맥 NC 이스트폴** 기준 기초자료를 단일 마스터로 삼음 (법적 사용 범위는 별도 확정) |

---

## 3. 세팅비 정책 (과금 명목)

아래 금액은 **서비스 이용료가 아니라 “초기 기초자료 세팅비” 명목**으로 받는 금액이다. 견적·계약서·결제 내역·안내 문구에도 **세팅비**로 통일하는 것이 좋다.

| 플랜 | 세팅비 | 설명 |
|------|--------|------|
| 표준 | **10만 원** | 고정 시드 소스 그대로 해당 테넌트에 적용 |
| 커스텀 | **20만 원** | 사용자가 원하는 분류·품목 구조에 맞춘 매핑/편집 후 적용 (워크플로 별도) |

- 결제는 기존 결제 수단(예: 토스페이먼츠)과 연동하고, **세팅비 결제·세팅 완료**를 테넌트 또는 주문/작업 테이블에 기록한다. (품목명 예: `초기 기초자료 세팅비 — 표준` / `— 커스텀`)
- 이벤트 시 **프로모 코드·크레딧·기간 한정 entitlement** 로 동일 기능을 무료 제공할 수 있음 (별도 정책).

## 4. 데이터 처리 규칙

### 4.1 자재 단가

- 시드 적용 시 **자재 단가(원가)는 전부 0원**으로 넣는다.
- 이후 **실제 매입·입고** 흐름에서 단가·재고를 갱신한다.
- 제품 측면: 원가 0 상태로 리포트가 나가지 않도록 **경고 배지·안내 문구** 검토.

### 4.2 거래처

- 기본 패키지에 **릴리맥 NC 거래처를 샘플**로 포함한다.
- 실사업자번호·개인 연락처 등 **민감 정보**는 저장소·시드 파일에 넣지 않거나, 더미+“수정 필요” 안내로 대체하는 방안을 택할 수 있음.

### 4.3 상품

- 이스트폴 기준 상품 목록을 시드에 포함하되, **단가·세금·바코드** 등 매장별 상이 필드는 정책에 따라 0/공란 또는 보수적 기본값 후 **매장 확인**을 전제로 한다.

### 4.4 카테고리

- `system_settings` 등 기존 구조에 맞춰 **상품·자재·지출** 카테고리 트리를 시드에 포함한다.
- 앱 내 기본 `DEFAULT_*_CATEGORIES` 와 병행할지, 시드가 **덮어쓰기**인지 **비어 있을 때만**인지 규칙을 명문화한다.

---

## 5. 시드 소스 “고정” 원칙

- **고정**이란: 매번 수작업 엑셀이 아니라, **단일 원본**(`seed/` 하위 JSON·CSV 또는 버전 태그된 스냅샷)을 두고 시드 작업은 **항상 그 원본만** 읽는다.
- **버전**: `v2026-04` 등 이름으로 스냅샷을 구분하고, 테넌트 적용 시 **어떤 버전을 썼는지** 로그에 남긴다.
- **민감 정보**: Git 공개 저장소에는 실거래 단가·실연락처를 올리지 않는다. 필요 시 비공개 저장소·암호화된 아티팩트·관리자만 접근 가능한 버킷을 사용한다.

---

## 6. 운영 프로세스 (안)

1. 고객이 **표준 세팅비(10만 원) 또는 커스텀 세팅비(20만 원)** 결제.
2. 커스텀인 경우 설문/엑셀 수령 → 담당자가 마스터에 매핑.
3. 관리자 도구에서 **대상 tenant_id 선택 → 시드 버전 선택 → 미리보기(건수·충돌)** → 실행.
4. 완료 후 고객에게 **“샘플·초안이므로 단가·거래처 확인”** 안내.
5. 이벤트 무료는 쿠폰/크레딧으로 동일 파이프라인 실행.

---

## 7. 기술 구현 방향 (코드베이스 연계)

현재 구조상 다음이 성립한다.

- 테넌트 단위: `products`, 자재, `suppliers`, `system_settings`(카테고리) 등 **tenant_id** 로 격리.
- **super_admin** RLS·정책 존재 → 서비스 롤 또는 관리자 전용 API로 대량 insert/upsert 가능.

구현 시 권장 사항:

- **Idempotency**: 동일 테넌트·동일 버전 재실행 시 중복 행 방지(자연 키 또는 `seed_batch_id`).
- **미리보기·diff**: 추가/스킵/충돌 요약.
- **롤백 또는 시드 전용 삭제**: 실수 복구 (범위 정의 필요).
- **감사 로그**: 실행자·시각·tenant_id·시드 버전.
